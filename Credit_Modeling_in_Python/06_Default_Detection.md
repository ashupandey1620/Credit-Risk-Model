### Understanding Default  
- **Definition of Default:** When a borrower fails to repay the loan installments (EMIs) on time.  
- Example: A borrower Rajat takes a house loan of ₹10 lakh from a bank (SBI) with a monthly EMI of ₹35,000.  
- If Rajat stops paying EMIs:  
  - After 1 month: He is recorded as **30 days delinquent**.  
  - After 2 months: **60 days delinquent**.  
  - After 3 months: According to **Basel guidelines** (a global regulatory body), Rajat becomes a **default customer** if delinquent for more than 90 days.  
- **Basel Committee** provides the global standards; 90 days past due threshold marks the transition from delinquency to default—a sign of potential inability to repay.  

[00:03:31]  
### Key Terminologies: Delinquency & DPD  
- **Delinquency** is tracked in the bank’s database in tables marking overdue payments.  
- **Days Past Due (DPD):** Industry-standard terminology representing how many days the payment is late:  
  | DPD Days | Description              | Interpretation                  |  
  |----------|--------------------------|--------------------------------|  
  | 30       | 30 days past due         | First month default/delinquency |  
  | 60       | 60 days past due         | Second month delinquency        |  
  | 90       | 90 days past due         | Default threshold by Basel     |  

- After 90 days DPD, a customer is **officially marked as default**.

### Default Detection Explained  
- Detection means **identifying the default event**.  
- Default detection involves **flagging defaults within the credit life cycle** and is essential to building credit risk models, as models rely on default data.  
- Without **default data**, credit risk modeling is impossible.  
- Hence, default detection is highlighted as **one of the most important steps** in credit risk management.

### Step 1: Adding Logic to the Performance File  
- The first step to default detection is adding **logic to a monthly performance file** to capture default triggering events and create a **default flag**.  

- **Performance File Definition:**  
  - A monthly dataset generated by the bank for every customer with a loan account.  
  - Contains **loan account payment data** and **customer characteristics**.  
  - Reflects the customer's **monthly repayment performance** and other financial attributes.  
  - This file includes all borrowers, not just defaulters.

- Typical fields in a performance file include:  
  - Salary  
  - Job type (government, private, business)  
  - Age  
  - Work experience (longer experience implies lower default risk)  
  - Status of loan payments (EMI paid/due)  

- **Event:** Occurs when a customer stops paying EMI (misses a payment), considered a triggering event for default detection.

### Examples of Default Triggering Events  
- Customer-related adverse events that trigger defaults include:  
  - Bankruptcy  
  - Business failure  
  - Job loss  
  - Death  

- These real-world triggers are integrated into the detection logic.  

- **Default Flag:**  
  - A binary column in the database indicating default status:  
    - 0 = Non-default  
    - 1 = Default (also called an event in modeling terms)

### Step 2: Creating Customer Level Default Episodes  
- The second step aggregates monthly account-level data into **customer-level default episodes** that span over a defined period.  
- The episode starts from the **event date** when the customer first defaults (e.g., stops paying EMI).  

- Example timeline:  
  | Date           | Event Description                           | Notes                 |   
  |----------------|---------------------------------------------|----------------------|  
  | July 6, 2025   | Customer misses EMI payment – event/starter | Episode start date    |  
  | July 2025 to   | Monitoring period                            | 12 to 18 months, varies by bank |  
  | July 2026      |                                            | Known as **cure period** or **cooling off period** |  

- The period from the first default to the end of the cure period (12–18 months) constitutes the **episode**.  

- If the customer pays again during this period but later defaults, a **new cooling off period** can start from the last default.  
  - For example, if default initially in July 2025 then paid in August and defaults again in October 2025, the new cure period starts October 2025.  
  - The full episode then extends from July 2025 to October 2026.  

- **Terminology to remember:**  
  - **Event**: Initial default occurrence (missed payment or adverse customer event)  
  - **Default Flag**: Binary indicator of default status  
  - **Episode**: The full cycle from default event until end of cure period/cooling off period  
  - **Cure Period (Cooling Off Period)**: Period during which no further default occurs, typically 12–18 months

### Types of Events Leading to Default  
- Various event types can lead to default classification:  
  - **Restructuring:** Customer’s loan terms are modified due to inability to pay. For example, reducing EMI to a manageable amount.  
  - **Charge Off:** Bank concludes there is a very low recovery probability (approx. 1% recovered), writing off the loan as a loss.  
  - **Bankruptcy:** The customer legally declares inability to repay debts.  
    - Example given: **Kingfisher Airlines** declared bankruptcy, resulting in government/federal interventions.  

- These events are critical components of the default detection system and impact the credit risk evaluation.
